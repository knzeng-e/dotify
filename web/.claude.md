# Dotify Context

Dotify is a decentralized cultural social hub, not only a music streaming app. Product, copy, and UI work should reinforce the idea that music can become a common layer for human connection and community coordination.

## Product Philosophy

- Dotify should feel like a web3 cultural experience: shared listening, artist-owned rights, direct support, portable metadata, and transparent access.
- Avoid positioning Dotify as a Spotify clone. Spotify can inspire density and player ergonomics, but Dotify's message is sovereignty, community, and protocol-backed culture.
- Primary UX language should make web3 values understandable without jargon: proof instead of profiles, wallets as keys, personhood as participation, open networks instead of platform lock-in.
- Authentication and trust moments should be short, centered, and reassuring. Emphasize no passwords, no platform login, and no personal data handover.

## Copy Principles

- Prefer user freedom over technical detail in primary labels.
- Use "Use wallet" or "Use my wallet" instead of generic "Connect wallet" when it improves clarity.
- Explain artist actions as ownership: artist-owned profile, catalog you control, publish under your own terms.
- Explain listener actions as agency: unlock with proof, support artists directly, join shared rooms.
- Keep protocol terms like EVM, IPFS, Bulletin, and personhood available in advanced or secondary context.

## Current Visual Direction

- White and midnight-blue product interface.
- Dense, practical layout inspired by Spotify's app ergonomics.
- Clean documentation-like surfaces inspired by Polkadot docs.
- Restrained cyan/blue accents, with Polkadot pink only where it adds meaning.

## Change Workflow

- After any file change, check the current `git diff` before summarizing the work.
- Always provide a detailed, ready-to-use commit message that reflects the actual diff.
- The commit message should mention both user-facing product changes and implementation details when both exist.
